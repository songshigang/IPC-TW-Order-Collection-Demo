2.3 Proposal Content

The proposal should contain at least the following information:

* Understanding of the Project
  + To demonstrate the vendor’s understanding of Kimberly-Clark business, project objectives and any potential issues during the implementation.
  + To demonstrate vendor’s insight in Kimberly-Clark business and also in Health and Hygiene care FMCG industry.
* Solution Overview
  + Vendor needs to explain the detailed technique solution design including but not limited to Product development technology/System architecture/System integration/Hosting/Data management and processing/Report and analysis/Security etc.
* Proposal on Implementation Approach
  + Vendor needs to explain the project schedule, project phasing and organization with profile of key project team members.
* Project Management Methodology
  + Vendor should explain how to execute the project including but not limited to, project management, change management, performance management and risk management etc.
* Project/Post-project Support Models and Service Levels
  + Vendor should propose and explain the project/post-project support models and service levels.
* Response to Functional Requirement
  + Vendors need to clearly identify which functionality exists in the current version of the proposed solution/product; which functionality needs to be customized / developed with the estimation of the effort (in man-days); which functionality is not possible to be implemented.
* Financial Proposal
  + Vendors need to provide their financial proposal including one-time implement cost and on-going cost.
* Customer Reference Cases
  + Vendors need to provide customer references with similar business nature of Kimberly-Clark such as Health and Hygiene care FMCG industry sectors.
* Opportunities for future extension
  + Vendors need to identify any areas of functionality that are not specifically included in this RFP and that it may offer significant benefit in future.
  + Kimberly-Clark expects that Vendors will provide additional information to clarify the submitted proposal, if appropriate.
  + Kimberly-Clark also expects the Vendors to determine its ability to be a strategic partner to Kimberly-Clark, as well as to propose any value added services relevant to this RFP.
* Successful use cases

1. Project Detail

**Background:**

台灣部分客戶有自有平臺（含寄售訂單形式），需要手工每日及月度登錄客戶平臺獲取相應訂單，再手工轉換格式生成訂單，因此希望這個平臺可以對接客戶訂單介面獲取訂單資訊，並轉換格式推送SAP生成EDI訂單

**Objective:**

開發KC自動化取單平臺方面主要功能，同時為確保業務正確運行需要例如：客戶主資料、物料主資料等輔助資料資訊的對接。

主要功能，

訂單資料抓取及轉單，帶\*為必須功能，其他依供應商開發能力提供

1. \*用戶端提供訂單資料介面，將訂單資料推送給KC訂單平臺（Daily，Monthly不同類型訂單格式PDF/Excel/HTML.TXT 等格式），預計22家客戶，假設匯入的檔案是PDF檔，需同步新增SAP單號在後續下載的PDF檔案上。
2. \*可以檢查訂單在創建後一周至1個月內是否有更改；可以by 客戶設置檢查週期
3. \*平臺訂單資料多條件查詢、下載
4. \*可以從該平臺創建並下載該訂單PDF檔（部分客戶需要由該平臺生成）
5. \*可以從 FTP 訂單資料夾或採集平臺下載 EDI 檔（排除重覆訂單，並換算成出貨箱數後，可以直接用於上傳SAP生成SO）
6. \*CSC 可以根據客戶，維護和管理可出貨產品及產品代碼、箱規資訊，可以批量上傳和下載
7. \*訂單中的樣品過濾；產品箱規檢查；重複訂單檢查; 不成箱檢查，客戶下單的單位並非都為CS) 、新品報錯
8. 自動關單設置；by customer/sold to/material/時間段等設置（替代SAP預設管理）-可選
9. \*可以維護客戶門店與 KC sold to和ship to的關係（GLN及閘店號）

其他，

1. 操作介面以中文繁體設置
2. 幫助文檔下載，幫助文檔即平臺操作說明的編寫
3. 使用者登錄及運算元據記錄
4. 用戶許可權維護與分配
5. 配合KC做相應認證工作及文檔提供，例如：供應商資訊安全認證等
6. 平臺供應商端相應工具證書費用。
7. 客戶平臺更換，或是格式調整，需協助維護
8. 支援增設後續新客戶抓取

**Phased approach proposal**

Phase 1 （本次招投標重點）:

滿足現階段已列出功能需求，保留更多資料介面，轉變人工作業為EDI訂單降低人工處理及溝通時間

Phase 2（未來需求展望）：

業務端收集使用者需求，結合已有介面及資料資源做更多定制化開發，不斷提升訂單處理及溝通效率
